## Context

当前管理端已迁移至 Arco 组件体系，但导航信息架构、列表检索能力与高频操作入口尚未与 v1 习惯及业务流程一致。此次变更同时覆盖前端导航壳层、列表页查询交互、后端列表查询参数、通用下拉组件与部分验收测试，属于跨页面、跨能力的体验一致性改造。

受影响范围集中在：
- 工作区壳层（header/sidebar/breadcrumb）
- 主数据与订单列表页（查询、表格列、排序、操作列）
- 订单详情页与订单列表页之间的删除入口职责
- 下拉检索与“使用当前输入”能力（商品明细编排链路）

## Goals / Non-Goals

**Goals:**
- 将菜单分组结构与面包屑映射调整为 v1 认知模型，降低路径学习成本。
- 为主数据与订单列表提供统一的多字段模糊搜索能力。
- 统一列表行为（创建/更新时间字段、更新时间倒序、备注文案、操作列紧凑且右侧固定）。
- 将“删除订单”统一收口到订单列表页，减少详情页高危操作。
- 恢复并完善系统下拉搜索与“使用当前输入”能力。
- 在 header 提供一键主题切换（参考 Arco Pro）。

**Non-Goals:**
- 不改动订单详情业务计算规则（金额计算、标红规则、导出逻辑）。
- 不引入新的后端服务或数据库表。
- 不重构现有鉴权与会话模型。

## Decisions

### Decision 1: 菜单与面包屑统一采用“配置驱动映射”
- 方案：定义一份导航树配置（含分组、路由、显示名、图标、是否占位页），sidebar 与 breadcrumb 共用该配置进行渲染与路径解析。
- 原因：避免 sidebar 与 breadcrumb 双份硬编码导致漂移。
- 备选：在两个组件各自维护映射；缺点是维护成本高、易不一致。

### Decision 2: 列表查询采用“单关键字 + 后端多字段 OR 模糊匹配”
- 方案：各列表接口增加 `q` 参数，由服务端将 `q` 作用到指定字段集合（`contains` + insensitive），并保持默认分页与排序行为一致。
- 原因：前端交互简单、后端约束明确、便于测试。
- 备选：前端多输入框高级筛选；对当前需求过重且改动面大。

### Decision 3: 列表通用行为通过页面约定统一
- 方案：除订单详情外，列表统一返回并展示 `createdAt`、`updatedAt`，默认按 `updatedAt desc`；字段文案统一为“备注”；操作列设置固定右侧并控制列宽。
- 原因：提升跨页面一致性，降低运营使用成本。
- 备选：各页独立定义；缺点是不可控且易分化。

### Decision 4: 订单删除入口迁移到订单列表
- 方案：订单详情页移除删除按钮，订单列表提供删除操作（含确认），并复用既有删除 API。
- 原因：高危操作与列表管理场景更匹配，减少详情误删风险。
- 备选：双入口并存；缺点是行为分散且增加误触点。

### Decision 5: 下拉搜索与“使用当前输入”以统一组件恢复
- 方案：在通用下拉组件中恢复搜索防抖、远程查询与“使用当前输入”首项逻辑，并确保明细表单分类/单位/商品链路一致生效。
- 原因：避免各页实现差异导致能力回退。
- 备选：页面内局部修复；缺点是重复实现、易再次丢失。

### Decision 6: 主题切换采用 Arco ConfigProvider + 本地持久化
- 方案：header 增加主题切换按钮，切换 `light/dark`，状态写入 `localStorage` 并在应用加载时恢复。
- 原因：与 Arco Pro 实践一致，成本低、收益直接。
- 备选：仅切换局部 CSS 变量；缺点是 Arco 组件与自定义样式不易一致。

## Risks / Trade-offs

- [Risk] 搜索字段扩展后列表查询压力上升  
  → Mitigation: 仅对必要文本字段做模糊匹配，保持分页，必要时为高频字段补索引。

- [Risk] 菜单重构引发路由/面包屑回归问题  
  → Mitigation: 使用单一配置源 + 增加导航与 breadcrumb 的 E2E 断言。

- [Risk] 删除入口迁移导致用户短期不适应  
  → Mitigation: 在订单详情页提供返回列表操作与文案提示，发布说明明确变更。

- [Risk] 主题切换与现有自定义样式冲突  
  → Mitigation: 清理历史定制样式，优先使用 Arco 主题 token 与组件默认样式。

## Migration Plan

1. 先交付导航配置与面包屑映射，确保路由与壳层稳定。
2. 再交付各列表后端搜索参数、默认排序与字段补齐，前端接入统一查询输入。
3. 调整列表列定义（备注文案、时间字段、操作列固定右侧）。
4. 迁移删除订单入口并移除详情页入口。
5. 修复下拉搜索与“使用当前输入”能力。
6. 接入 header 主题切换并做持久化。
7. 补齐 API/E2E 测试与文档。

回滚策略：
- 导航与列表查询改动均可按页面/接口粒度回滚；
- 删除入口迁移可在必要时临时恢复详情页按钮；
- 主题切换可通过隐藏开关快速降级。

## Open Questions

- “工作台（页面占位）”是否需要在本次提供最小可见占位内容，还是仅保留可访问空壳页。
- 列表时间字段展示格式是否统一为本地时间（例如 `YYYY-MM-DD HH:mm:ss`）并固定时区。
- 订单列表中的“进货地/市场名称”搜索是否需要拆分为两个独立条件（当前提案为单关键字匹配）。
