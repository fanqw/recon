## Context

本次是在同一变更内统一完成导航、搜索、列表规范、删除入口迁移、主题切换、下拉可用性修复与测试数据基线标准化。变更跨壳层组件、多个列表页、订单详情页、API 查询与 seed 脚本，需保证规则一致与回归可验证。

## Goals / Non-Goals

**Goals:**
- 菜单结构与面包屑对齐 v1 信息架构。
- 完成主数据与订单列表的多字段模糊搜索。
- 统一列表行为（创建/更新时间、更新时间倒序、备注文案、紧凑右侧固定操作列）。
- 删除订单入口迁移至订单列表，详情页移除高危入口。
- 侧边栏支持收起/展开。
- header 主题切换固定为 light/dark，并持久化。
- 恢复下拉搜索与“使用当前输入”。
- seed 数据清库后回填语义化中文样本，用于测试。
- 订单详情页详情区紧凑化，提升明细可视面积。

**Non-Goals:**
- 不改动订单金额计算、导出与明细业务规则。
- 不新增数据库表结构。
- 不改动鉴权与会话模型。

## Decisions

### Decision 1: 菜单与面包屑采用单一配置源
- 方案：定义导航树配置（分组、路由、文案、图标），侧栏与面包屑共用。
- 原因：避免壳层映射分散导致不一致。

### Decision 2: 列表搜索采用统一 `q` 参数
- 方案：列表 API 接收 `q`，后端对约定字段执行 `contains + insensitive` OR 模糊匹配。
- 原因：前端交互简洁，后端行为统一可测试。

### Decision 3: 列表规范统一落地
- 方案：除订单详情外，所有列表返回并展示 `createdAt/updatedAt`，默认 `updatedAt desc`；“描述”统一为“备注”；操作列固定右侧且紧凑。
- 原因：跨页面一致，降低使用成本。

### Decision 4: 删除入口收口到订单列表
- 方案：详情页删除按钮移除；列表操作列新增 danger 删除按钮 + 确认交互。
- 原因：高危操作集中管理，降低误删。

### Decision 5: 侧栏收起与主题切换由壳层状态管理
- 方案：壳层持有 `collapsed` 与 `theme` 状态；主题仅支持 `light/dark` 并写入本地存储。
- 原因：状态集中，行为稳定。

### Decision 6: 下拉搜索与“使用当前输入”走通用组件
- 方案：统一下拉组件恢复搜索、防抖、候选刷新与“使用当前输入”首项逻辑。
- 原因：避免页面级补丁反复失效。

### Decision 7: seed 提供中文语义化可重复基线
- 方案：seed 执行“清空业务数据 -> 重建分类/单位/商品/进货地 -> 创建示例订单与明细”，样本采用约定中文语义。
- 原因：提高联调可读性和自动化稳定性。

### Decision 8: 订单详情信息区改为紧凑布局
- 方案：顶部区域改为紧凑信息条/双栏布局，主操作与关键字段聚合展示。
- 原因：减少垂直占用，优先展示明细表。

## Risks / Trade-offs

- [Risk] 搜索字段扩展增加查询压力
  → Mitigation: 保持分页、仅对必要字段模糊匹配，必要时补充索引。

- [Risk] 主题初始化闪烁
  → Mitigation: 提前读取本地主题并尽早应用。

- [Risk] 清库 seed 覆盖本地调试数据
  → Mitigation: 文档明确 seed 会重置业务数据。

- [Risk] 删除入口迁移造成短期习惯变化
  → Mitigation: 详情页保留“返回列表”引导，发布说明强调入口变化。

## Migration Plan

1. 壳层重构：菜单结构、面包屑、侧栏收起、light/dark 切换。
2. 列表能力：搜索、时间字段、默认排序、备注文案、操作列固定。
3. 订单改造：删除入口迁移 + 详情区紧凑化。
4. 下拉能力修复：全局搜索与“使用当前输入”。
5. seed 改造：清库并写入中文语义化样本。
6. 验收与文档：补齐测试与 README。

回滚策略：
- 壳层与列表改动可按页面分批回滚。
- 删除入口可应急恢复详情页按钮（临时）。
- seed 可回退至旧版本脚本。
