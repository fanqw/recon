## 上下文

v2 应用已经具备首版数据统计所需的数据基础：订单关联进货地，订单明细关联商品，商品关联分类与单位，订单明细通过 `line_total` 持久化业务金额，且订单详情页已有基于该金额的合计语义。当前 `/workspace` 页面仍是占位页，而导航规范已经将“工作台”作为工作区一级入口。

本次变更需要把占位入口升级为只读数据工作台，不改变现有订单录入流程，也不引入新的账务流水或库存估值表。首版统计应完全来源于已有未删除数据。

## 目标与非目标

**目标：**

- 提供数据工作台，按分类、商品、进货地与时间维度汇总订单明细金额。
- 使用 `OrderCommodity.lineTotal` 作为默认金额来源，与订单详情和导出语义保持一致。
- 支持按天、周、月、年查看总金额趋势。
- 将聚合逻辑放在图表组件之外，便于单元测试。
- 在无数据或数据稀疏时仍提供可理解的页面状态。

**非目标：**

- 不新增财务记账、结算、对账或库存估值模型。
- 不在本次变更中提供用户自定义仪表盘能力。
- 不在图表内提供下钻编辑；源数据仍通过现有订单与主数据页面维护。
- 不要求回填或迁移现有持久化数据。

## 技术决策

### 服务端聚合契约

统计数据应由服务端加载器或 API 路由通过 Prisma 查询和确定性的转换函数生成。客户端应接收可直接用于图表和表格的数据序列，而不是拉取全量订单明细后自行聚合。

已考虑方案：

- 客户端加载全部明细后聚合：初期实现简单，但数据增长后页面性能风险较高，也容易复制业务口径。
- 数据库 SQL 视图：效率较高，但首版演进成本更重，也不如当前 Prisma 为中心的应用结构一致。

### 金额与删除语义

所有总金额应汇总满足 `OrderCommodity.deleted = false` 且父级 `Order.deleted = false` 的 `OrderCommodity.lineTotal`。已逻辑删除的维度数据不应再作为新的筛选选项出现，但历史订单明细仍可按其关联名称参与聚合展示。

这一口径与现有订单详情契约一致：分类金额和订单金额均基于 `line_total`，而不是重新计算 `price * count`。

### 时间维度来源

趋势时间桶首版应使用 `Order.createdAt` 作为业务日期来源，因为当前模型没有独立的进货日期、到货日期或记账日期。这样可以避免引入含义不明的日期口径。

默认时间范围应展示最近一个月的数据，页面必须让用户能识别当前生效范围。周统计以周一作为每周起始日，所有周时间桶和标签都应按这一规则生成。

如果未来新增明确的业务发生日期，可通过新的规范变更切换时间桶来源，并补充迁移说明。

### 图表组成

工作台图表应直接对应用户要回答的问题：

- ECharts 柱形图用于分类、商品、进货地金额排行或对比；商品金额图展示金额排行 Top 50，避免商品过多导致图表不可读。
- ECharts 折线图用于按天、周、月、年的金额趋势。
- ECharts 饼图分别用于分类金额占比和进货地金额占比，首版同时保留这两个占比图。
- 指标摘要用于展示总金额、订单数、明细数和平均订单金额。

实现阶段应使用 ECharts 作为图表库，并选择兼容 Next.js App Router 客户端组件的集成方式。ECharts 渲染应隔离在客户端组件内，领域统计契约不应依赖 ECharts option 结构。

### 筛选与空状态

首版工作台应包含时间范围筛选和时间粒度选择。默认时间范围为最近一个月，维度筛选可作为可选增强，但不应阻塞核心工作台交付。无数据时应展示明确的空状态和零值指标，而不是展示损坏图表或旧数据。

## 风险与权衡

- 周统计边界在不同地区可能理解不同。缓解：规范固定为周一开始，并通过测试和可见标签明确表达。
- Decimal 金额过早转为 number 可能带来精度问题。缓解：聚合阶段使用 Decimal 或数据库 numeric 值，只在响应或展示边界转换。
- 图表库若依赖 `window`，可能导致 SSR 或 hydration 问题。缓解：将图表渲染隔离到客户端组件，聚合逻辑不依赖图表库。
- 大数据量后可能需要索引或分页策略。缓解：首版先服务端聚合，并在查询层支持日期过滤，后续按性能指标优化。

## 迁移计划

不需要数据库迁移。本次变更可以作为增量 UI/API 能力发布。回滚时可隐藏工作台路由实现或回退导航行为；已有数据不受影响。

## 已确认问题

- 默认时间范围：最近一个月。
- 周统计起始日：周一。
- 图表库：ECharts。
- 商品图：展示金额排行 Top 50。
- 饼图：同时保留分类占比和进货地占比。
