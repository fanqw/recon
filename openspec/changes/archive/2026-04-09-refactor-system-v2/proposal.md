# Proposal：refactor-system-v2

## Why

v1 采用前后端分离（Express + 独立 React 客户端），与账本/对账业务相关的代码分散、类型与交付流程不统一，长期维护成本高。需要在保留 **v1 既有业务语义** 的前提下，用现代全栈栈（Next.js、Prisma、TypeScript）重建系统，并以 **空库上线、直接替换** 的方式完成切换，便于统一规范、测试与容器化交付。

## What Changes

- 在 mono-repo 中落地 v2 **全栈 Web 应用**（`apps/web`），承载页面、服务端接口与数据访问编排；**不**再维护与 v1 并行的生产双栈。
- 使用 **PostgreSQL + Prisma** 定义数据模型与迁移；**空库启动**，不要求从 v1 数据库迁移历史数据。
- 实现与 v1 对齐的 **页面级业务能力**：登录、分类、单位、商品、订单列表与订单详情（含订单明细逻辑），**仅保留原始业务逻辑**，界面可按新项目重做。
- 鉴权模型为 **单一管理员角色**（已登录即具备业务操作权限；未登录访问受保护资源应被拒绝或重定向）。
- **BREAKING**：不保证与 v1 REST 路径、响应体或 Cookie 形状兼容；验收以 **页面行为与业务规则** 为准，而非旧接口契约。
- 引入 **自动化验收**：覆盖核心 API 可用性及关键页面可操作路径（如按钮可点、主流程可走通），纳入可重复执行的测试命令（供本地与 CI 使用）。

## Capabilities

### New Capabilities

- `admin-authentication`：管理员登录、登出、会话与受保护路由/API 的未授权行为。
- `master-data`：分类、单位、商品的列表与增删改查及其与 v1 一致的业务关联（如商品依赖分类与单位）。
- `sales-orders`：订单与订单明细的列表、详情、创建/维护流程及与主数据的业务关系。
- `automated-acceptance`：针对上述能力的 API 层测试与浏览器端（E2E）冒烟/主流程测试要求及运行约定。

### Modified Capabilities

- （无）当前 `openspec/specs/` 下尚无已发布能力规格。

## Impact

- **产品名称**：交付与文档中的 v2 项目统一称为 **recon**（包名、Cookie 前缀、数据库名等实现细节与之一致或以其为前缀）。
- **新增**：`apps/web`、`packages/*`（若采用共享包）、Prisma schema 与迁移、Docker Compose 与 `.env` 约定、测试目录与脚本。
- **基础设施**：PostgreSQL、Redis（会话或缓存等按设计使用）；开发/CI 需能启动依赖并执行 `db:migrate` 与测试。
- **参考**：`v1/` 仅作对照与需求核对，不作为运行依赖。
- **文档**：`README.md` / `AGENTS.md` 中与目录、命令、环境变量相关的条目需随实现更新。
