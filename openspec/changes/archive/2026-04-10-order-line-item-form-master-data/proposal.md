## Why

当前新建订单明细仅支持选择已存在的商品 id，无法在表单中同时维护分类、单位与商品名称的连贯体验，也无法在库中无完全匹配项时用关键字快速落库主数据。业务需要与 v1 类似的行级录入效率，并支持行金额的手动校正；标红通过 **`lineTotal` 与舍入基准 `total_price`（单价×数量按约定舍入）是否一致**判定，无需额外布尔列。

## What Changes

- 新建订单明细表单增加**分类**、**单位**可搜索下拉；选择商品后自动带出商品关联的分类与单位作为默认值。
- 商品、分类、单位三个控件均支持**关键字过滤**；当输入文本无法与现有选项完全匹配时，在下拉**首项**展示「使用当前输入」类选项，用户可选中提交。
- 提交时服务端**按序解析或创建**：分类 → 单位 → 商品（商品由名称 + 分类 id + 单位 id 定位；已存在则复用），再创建订单明细关联该商品 id；主数据名称**持久化前 `trim()`**。**`trim()` 后为空**：**页面先提示并拦截提交**，**接口再二次校验**返回 400，不落库。
- 增加持久化 **`lineTotal`**：默认与 `total_price`（`lineRoundedTotal`）一致；用户手动修改时**原样存储、不做舍入**。**不增加 `amountAdjusted`**；标红 = `lineTotal` ≠ 舍入基准 `total_price`。
- **分类合计、订单总合计**对各行 **`lineTotal` 求和**。
- 订单明细列表、详情表格与 Excel 导出中金额列标红**仅依据**上述比较，不再用 `total_price` 与 `origin_total_price` 的差。
- **Category / Unit / Commodity**：名称在 **`trim()` 后**参与唯一性；**Category、Unit** 在未删除行上对 **name** 部分唯一；**Commodity** 在未删除行上对 **(name, category_id, unit_id)** 部分唯一（数据规则，与示例名称无关）。  
- 种子与 mock 采用**贴近真实业务**的示例类型（如分类「水果」、单位「斤」、商品「苹果」），便于演示；**不**将「水果/斤/苹果」作为唯一索引的设计动机。
- **旧订单行数据**：本变更**不要求**为历史行回填或兼容 `lineTotal`（按新库或清空策略处理）。

## Capabilities

### New Capabilities

（无独立新能力目录；行为增量落在下列已存在能力中。）

### Modified Capabilities

- `sales-orders`：订单明细创建流程、列表/表格字段与标红规则、导出规则、聚合口径与 `lineTotal` 相关需求。
- `master-data`：名称 trim、未删除行 name 唯一、列表检索、商品三元组复用。

## Impact

- **数据模型**：`OrderCommodity` 增加 `lineTotal`；`Category`/`Unit`/`Commodity` 增加与 trim 后名称一致的部分唯一索引；**不做**历史订单行回填。
- **API**：订单行创建/列表/更新契约与聚合逻辑调整；主数据创建需遵守唯一约束。
- **前端**：表单、表格、导出与 `lineTotal` / 舍入基准比较一致；**名称 trim 后为空**在**页面上提示并拦截提交**（与后端 400 二次校验配套）。
- **种子数据**：示例主数据改为贴近业务的名称（水果 / 斤 / 苹果），与唯一索引条款相互独立。
