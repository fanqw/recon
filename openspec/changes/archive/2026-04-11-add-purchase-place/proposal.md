## Why

当前订单仅包含名称与备注，缺少结构化的进货来源信息，导致下单与后续追溯依赖自由文本，数据不可复用且难以校验。现在需要在保持现有主数据与软删除语义的前提下，引入“进货地”并让订单强制关联，统一录入口径与删除约束。

## What Changes

- 新增“进货地”主数据模块，字段包含：进货地、市场名称、备注，支持与分类/单位一致的列表与 CRUD（逻辑删除）。
- 新增未删除数据上的组合唯一约束：`进货地 + 市场名称`，避免同一来源重复建档。
- 订单创建与更新改为必须关联进货地（`purchasePlaceId`），并在订单页面展示关联信息。
- 删除规则收敛为“存在未删除关联记录时不可删除”：
  - 分类被未删除商品引用时不可删除；
  - 单位被未删除商品引用时不可删除；
  - 商品被未删除订单明细引用时不可删除；
  - 进货地被未删除订单引用时不可删除。
- 前端在删除动作中给出明确提示，后端接口返回冲突错误并兜底拦截，避免绕过前端直接删除。

## Capabilities

### New Capabilities

- 无

### Modified Capabilities

- `master-data`: 扩展主数据范围至进货地，并新增“被未删除记录关联时不可删除”的删除约束。
- `sales-orders`: 订单模型与创建/更新流程新增“必须关联进货地”的业务要求。
- `automated-acceptance`: 自动化验证范围扩展到进货地与删除拦截场景，覆盖 API 与关键页面流程。

## Impact

- 数据模型与迁移：`apps/web/prisma/schema.prisma` 与新增 migration（进货地表、订单外键、组合唯一索引）。
- 后端 API：新增进货地 API；调整订单 API 入参与返回；调整分类/单位/商品删除接口的关联拦截逻辑。
- 前端页面：新增进货地管理页与导航入口；订单新建/编辑与详情展示增加进货地字段。
- 测试：更新 API 集成测试与 E2E 覆盖，新增进货地 CRUD、订单必填关联、删除冲突拦截用例。
