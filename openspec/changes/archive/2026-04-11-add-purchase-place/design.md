## Context

当前 v2 使用 Next.js + Prisma + PostgreSQL 单体全栈结构，主数据（分类、单位、商品）与订单/订单明细均采用逻辑删除语义。现状中订单仅包含 `name` 与 `desc`，缺少结构化进货来源；同时主数据删除接口尚未对“被未删除业务数据引用”进行拦截，存在删除后业务语义破坏风险。

本次变更需要同时覆盖数据模型、API、前端页面、测试与规范，属于跨模块改动，并且包含数据库迁移与种子数据调整。

## Goals / Non-Goals

**Goals:**
- 引入“进货地”主数据，字段为进货地、市场名称、备注，保持与现有主数据一致的管理体验与逻辑删除语义。
- 订单创建/更新必须关联进货地，形成稳定的订单头部来源信息。
- 建立统一删除保护：仅当存在未删除关联记录时禁止删除分类、单位、商品、进货地。
- 在数据库层保证进货地的未删除组合唯一（进货地 + 市场名称），并在 API 层返回明确冲突错误。
- 通过 API 与 E2E 自动化覆盖上述关键路径，防止回归。

**Non-Goals:**
- 不改变订单明细聚合、金额计算、导出 Excel 的现有规则。
- 不引入多租户、组织级隔离或跨仓库主数据共享能力。
- 不在本次变更中重构现有主数据页面为通用元编程页面。

## Decisions

### 1) 新增独立 `PurchasePlace` 实体并由订单外键引用

- 决策：在 Prisma 中新增 `PurchasePlace` 模型，并在 `Order` 增加 `purchasePlaceId` 外键关联。
- 原因：进货地是可复用主数据，不应被订单备注替代；外键关联可防止无效引用并支持后续筛选统计。
- 备选方案：
  - 仅在订单存储字符串字段（如 `sourcePlace/sourceMarket`）：实现简单但无法复用、难去重、难约束。
  - 将进货地并入商品：不符合“订单头来源”语义，且同商品跨订单来源可能不同。

### 2) 订单侧采用“必填关联”，历史数据按环境清空后重建 mock

- 决策：`purchasePlaceId` 在业务层与持久化层均按必填落地；现有历史订单数据采用清空策略，不执行回填脚本；开发/演示 mock 数据使用语义化中文。
- 原因：已确认可清空历史数据，回填复杂度与误配风险无收益。
- 备选方案：
  - 回填占位值：可保留历史记录，但会引入“占位值污染业务含义”问题。
  - 永久允许空值：与“必填”业务要求冲突。

### 3) 进货地主数据唯一性使用“未删除行上的组合唯一索引”

- 决策：建立部分唯一索引（`placeName + marketName`，`deleted=false`）。
- 原因：与现有分类/单位/商品唯一策略一致，支持软删除后重建同名记录。
- 备选方案：
  - 全表唯一（含已删除）：会阻止软删除后重建，不符合当前仓库语义。
  - 仅按进货地名唯一：无法区分同地不同市场，不满足业务判重口径。

### 4) 删除保护由“前端提示 + 后端兜底”双层实现

- 决策：删除前端给出明确失败提示；后端 `DELETE` 统一执行关联检查并返回冲突响应（409）。
- 原因：前端可提升可解释性，后端可防止绕过 UI 的非法删除。
- 备选方案：
  - 仅前端判断：可被直接调用 API 绕过。
  - 仅依赖数据库外键 RESTRICT：错误信息不可控，且当前为软删除模型，不直接适配。

### 5) 关联检查口径统一为“未删除关联记录”

- 决策：仅 `deleted=false` 的关联记录会阻止删除。
- 原因：与当前软删除默认查询口径一致，避免“历史已删除关联导致永久不可删”。
- 备选方案：
  - 任意历史关联均阻止：会造成主数据长期无法治理，运维成本高。

### 6) 删除失败响应采用稳定错误码枚举

- 决策：删除拦截返回统一结构，包含错误码枚举（如 `CATEGORY_IN_USE`、`UNIT_IN_USE`、`COMMODITY_IN_USE`、`PURCHASE_PLACE_IN_USE`）与展示文案；前端基于错误码映射提示。
- 原因：避免仅依赖中文消息字符串导致的脆弱匹配，便于测试稳定断言。
- 备选方案：
  - 仅返回中文消息：实现快，但前端逻辑与自动化断言易受文案改动影响。

## Risks / Trade-offs

- [历史数据清空导致联调样本不足] → 在种子脚本补充语义化中文 mock 数据，覆盖分类/单位/商品/进货地/订单最小链路。
- [删除检查分散在多个路由，行为不一致] → 在变更中统一错误语义与状态码，并补充 API 集成测试矩阵。
- [新增订单必填字段后，旧调用方请求体不兼容] → 明确返回 400 与错误文案，前端页面同次改造并更新文档。
- [组合唯一的 trim 口径不一致导致“看似重复却可写入”] → 在 API 层统一 trim 后写入，测试覆盖前后空白场景。

## Migration Plan

1. 新增 `PurchasePlace` 表与对应 API/页面，并建立订单外键关系。
2. 清空现有历史业务数据后重新写入语义化中文 mock 数据。
3. 前后端切换到“订单必填进货地”校验与提交流程。
4. 删除接口统一输出错误码枚举与文案，前端按错误码映射提示。
5. 补齐 API 与 E2E 自动化测试并更新文档。
6. 若回滚：保持数据清空后的最小 mock 集，撤回应用层必填与删除拦截改动。
