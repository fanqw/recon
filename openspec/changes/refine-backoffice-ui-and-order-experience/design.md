## Context

当前后台已经具备登录、工作区导航、主数据管理、订单管理和基础种子数据，但这些能力由多次迭代逐步叠加，视觉和交互的一致性不足。现状主要问题有五类：

1. 登录页虽然已具备基础布局，但视觉层次仍弱，品牌与业务语义不够明确。
2. 工作区壳层存在细节不协调，尤其是侧边栏顶部收起区与菜单之间的空白、以及面包屑层级在不同页面下不完全贴合菜单结构。
3. 主数据列表页和订单列表页的工具区模式不统一，卡片标题、搜索框、新建按钮的空间关系不稳定。
4. 表单必填项目前主要依赖原生 `required` 与普通标签，缺少统一的必填标识、校验反馈位置和视觉约束。
5. 订单详情页的“总金额”规则在领域上已部分具备，但页面交互仍未完整体现“默认自动计算 + 允许手动覆盖”的业务意图；同时信息区与表格区的视觉组织还可以进一步收敛。

这次变更横跨登录页、工作区壳层、主数据列表、订单列表、订单详情和 seed 数据，属于典型的跨模块体验收敛变更。它不引入新的业务实体，但会同时影响 UI 结构、表单行为、接口契约和联调数据。

## Goals / Non-Goals

**Goals:**

- 在不改变既有登录鉴权语义的前提下，重新设计登录页视觉层次与内容编排。
- 让侧边栏顶部与菜单区连续衔接，避免收起按钮旁的留白破坏整体节奏。
- 让面包屑严格反映当前菜单组、列表页和详情页层级。
- 统一系统表单必填项的表现方式，让主数据、订单、登录相关表单呈现一致的必填标签和错误提示。
- 统一主数据与订单列表页的工具区布局，减少重复标题，提升扫描效率。
- 明确订单详情页总金额字段的计算、手动覆盖和持久化规则，并同步页面交互。
- 用更真实的中文 mock 数据替换当前偏自动化样例的业务数据，便于联调和演示。

**Non-Goals:**

- 不新增新的业务模块或菜单层级。
- 不改变登录接口、会话机制、权限模型。
- 不重构整个设计系统，也不引入全新的 UI 组件库。
- 不修改订单聚合报表或工作台统计口径。
- 不新增数据库 schema；本次数据调整仅通过 seed 重建业务样本。

## Decisions

### 1. 登录页采用“语义化视觉区 + 表单区”的双区布局，但不改变登录接口

登录页继续复用现有 `/api/auth/login` 和会话逻辑，只重构页面编排与样式层。视觉上采用更强的采购/台账语义内容区，表单区保留明确的登录动作和错误反馈，避免把这次变更扩大为认证流程重写。

已考虑方案：

- 仅调整卡片配色和标题文案：改动小，但无法解决“整体单调”的核心问题。
- 引入完整营销式 landing hero：视觉更强，但不符合后台系统首屏定位，也会增加无效信息。

最终选择保留后台入口的工具属性，用更有层次的视觉区增强识别度，同时让表单仍然是首要操作区。

### 2. 工作区壳层以“空间修正”方式优化，不引入新导航模式

侧边栏继续沿用当前分组菜单和收起/展开机制，只调整顶部收起按钮容器、菜单起始位置和间距规则，避免按钮下方形成空白孤岛。面包屑继续基于现有 `workspace-nav` 生成，但重写路径映射规则，使列表页、详情页和所属菜单组之间的层级始终一致。

已考虑方案：

- 彻底改为顶部导航：影响范围过大，且与当前后台结构不匹配。
- 仅微调 CSS 间距：可以解决空白，但无法系统解决面包屑层级偏差。

因此本次保持导航形态不变，重点修正布局和层级推导逻辑。

### 3. 列表页统一为“卡片内工具条 + 表格内容区”的结构

商品分类、商品单位、商品信息、进货地和订单列表页面统一取消卡片标题，将搜索框和“新建”按钮放在卡片内部首行，通过 `space-between` 对齐。这样可以减少页面重复标题占位，并将用户最常用的搜索与新增动作收敛到同一视线带。

已考虑方案：

- 保留卡片标题，仅把按钮移入内容区：仍会形成标题、工具条、表格三层堆叠，信息密度较低。
- 完全移除卡片：部分页面会显得表格直接贴在页面背景上，观感不稳定。

最终统一为“无卡片标题，但保留卡片容器”的方案，兼顾结构稳定与视觉收敛。

### 4. 必填项体验抽象为共享表单约定，而不是散落在各页单独处理

本次会新增一层共享表单约定，统一必填标签样式、必填星号位置、错误文案展示位置和触发时机。页面仍可使用现有 Arco 表单控件或现有表单写法，但用户可感知的规则要一致，避免有的页面只靠浏览器拦截、有的页面靠文本错误提示。

已考虑方案：

- 只在每个页面手动补红色星号：实施快，但规则容易再次分散。
- 全量切换到 Arco `Form` 重构所有表单：一致性最好，但本次范围过大。

因此采用“共享视觉与交互约定 + 渐进接入”的方案，优先覆盖本次涉及页面。

### 5. 订单详情总金额沿用现有 `lineTotal` 持久化语义，并补齐“自动计算 + 手动覆盖”交互

领域上继续以 `lineTotal` 作为持久化行金额。UI 层把字段文案统一为“总金额”，默认值在数量和单价变化时按 `数量 * 单价` 计算，最多保留两位小数；当用户手动修改后，提交值按用户输入持久化。为避免用户输入被反复覆盖，前端需要区分“自动联动状态”和“已手动覆盖状态”。

已考虑方案：

- 始终强制总金额等于数量乘单价：简单，但不符合用户已明确提出的手动修改需求。
- 完全取消自动计算，仅由用户输入：会增加录入负担，也与现有业务习惯不符。

最终保留默认联动并支持手动覆盖，和当前领域模型保持一致。

### 6. 种子数据改为“可重建的真实业务样本”，继续通过现有 seed 入口交付

Prisma seed 仍然负责清理业务数据并重建样本，但样本内容会从“测试导向的最小可用数据”升级为“更接近真实采购场景的中文 mock 数据”，包括更完整的主数据、更自然的订单命名、更分散的数量与金额分布，以及可直接用于列表页、详情页、工作台联调的数据。

已考虑方案：

- 额外新增一套 demo seed：会引入多套数据口径，维护成本高。
- 保留当前 seed 不动，仅在测试里造数据：不利于本地体验和产品验收。

因此继续使用单一 seed 入口，但重写数据内容和组织方式。

## Risks / Trade-offs

- [风险] 登录页视觉增强后，移动端可能出现信息区过高或首屏被表单挤压。  
  → 缓解：设计要求中明确桌面与平板优先保持表单首要可见，移动端允许视觉区缩短或下移。

- [风险] 必填项体验统一若只做视觉层改造，仍可能保留部分页面原生校验时机不一致。  
  → 缓解：在规格中同时约束标签表现与错误提示触发，任务中覆盖共享抽象和页面回归。

- [风险] 订单总金额支持手动覆盖后，数量或单价再次变化时，是否继续联动可能产生歧义。  
  → 缓解：明确默认联动仅在用户未手动编辑前生效；一旦用户手动修改，总金额字段进入手动模式，除非用户主动重置。

- [风险] 列表页全部去掉卡片标题后，页面信息来源可能变弱。  
  → 缓解：依赖菜单高亮、面包屑和工具条文案维持上下文识别，不再重复冗余标题。

- [风险] 更真实的 seed 数据若与现有测试断言绑定过深，可能导致测试不稳定。  
  → 缓解：测试继续断言稳定结构和关键字段，不依赖过细的文案偶然值；如需固定断言，集中维护约定样本名称。

## Migration Plan

1. 先更新 OpenSpec 规格，明确登录页、导航、表单、订单与种子数据的新行为。
2. 实现共享表单必填项约定和各页面 UI 调整，保证列表页与订单详情页布局同步落地。
3. 调整订单明细新增/编辑逻辑，补齐总金额自动计算与手动覆盖的前后端契约。
4. 重写 Prisma seed，保证执行后清空旧业务数据并生成新的真实 mock 数据。
5. 通过单元测试、API 测试、E2E/验收路径和 `openspec validate` 验证变更。

回滚策略：

- UI 相关问题可直接回退页面组件和共享样式修改；
- 总金额交互若出现回归，可回退到当前 `lineTotal` 自动填充逻辑；
- 种子数据问题可通过回退 `prisma/seed.ts` 恢复旧样本，不涉及数据库结构迁移。

## Open Questions

- 登录页是否需要引入真实图片素材，还是仅基于现有设计系统做无图语义化重排；实现阶段可结合仓库现有前端规范再定。
- 订单总金额进入“手动模式”后，是否需要显式提供“恢复自动计算”入口；当前提案默认不强制要求，只保证手动值不会被隐式覆盖。
